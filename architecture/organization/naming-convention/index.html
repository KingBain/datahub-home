Naming Convention The name of a cloud resource is crucial for effective identification. Establishing a consistent naming convention across cloud providers enhances clarity and resource management. The Aurora team follows a naming convention with these key goals:
  • Indicate resource use and purpose: Resource names should clearly indicate their intended use to prevent accidental actions and enable quick location.
  • Enable sorting and filtering: Resource names should support easy sorting and filtering across cloud platforms.
  • Ensure uniqueness: Each resource name should be unique within its enterprise tenant to avoid conflicts and confusion.
  • Adhere to provider-specific naming restrictions: Resource names must follow the specific naming constraints set by each cloud provider:

Fields A common field approach has been leveraged to align with the goals of the naming convention, including indicating the purpose of the resource, facilitating filtering and sorting, and striving for uniqueness within the respective tenant. The following common fields can be included in a resource name:
  • User-defined (optional): This field can be used to help identify the purpose of the resource. For example, the value for this field may be “velero” for a Velero storage account.
  • Project: The project the resource belongs to. An acronym or short-form version of the project name should be used if possible.
  • Environment: The stage of the development lifecycle that the resource belongs to (ex: dev, test, qa, and prod).
  • Region: The region where the resource is deployed. The value of this field shall be the region’s acronym to reduce the number of characters used. For example, if the resource is deployed in Canada Central, the value for the region field shall be “cc.”
  • Instance: The instance count is for a specific resource to identify more than one resource that has the same naming convention. Examples are “01” and “02”. An example of a use case where this might be useful is if a series of VMs with the same purpose is required.
  • Resource Type: An abbreviation that represents the resource or asset type.
For consistency and readability, the fields should be separated by hyphens, and all characters should be lowercase. It is important to mention that some resource types may not have all of the above fields (referred to as the common naming convention). Scope Level Naming Inheritance When organizing resources within containers such as resource groups, there are two approaches to naming inheritance: explicit and implicit.
  • Explicit Inheritance: With explicit inheritance, child resources are named using the same common fields as their parent resource. This approach ensures consistency and precise identification of resources within the container. By inheriting the naming convention from the parent, it becomes easier to understand the purpose and context of the child’s resources.
  • Implicit Inheritance: Unlike implicit inheritance, child resources do not necessarily have the same fields in their names as the parent resource. This approach reduces the character length of resource names, mainly when dealing with resource types with long names. Implicit inheritance can minimize the size of resource names while maintaining some level of organization and identification within the container.
The decision on whether to use explicit or implicit naming inheritance at each scope (subscription, resource group, and resource) should be based on the organization’s and project’s specific requirements and considerations. Factors such as the desired level of consistency, the need for precise identification, and the impact on character length should be considered. 2024-10-21
